### 2. Core Approach to Metadata Aggregation/Segregation

IP-freely uses a **graph-based relational model** to handle metadata inheritance:

#### Two-way Association Mapping:
- **m4d (Metadata-for-Data):** Maps each data file to its associated metadata files, indexed by extension (`.json`, `.bval`, `.bvec`, `.tsv`)

### Package Information

**Current Version:** 1.2.2 (as of March 2026)  
**Package Name:** `bidsschematools`

### Core API Overview

#### **Primary Entry Point: `load_schema()`**

```python
from bidsschematools import schema

# Load default bundled schema
schema_obj = schema.load_schema()

# Load from custom YAML directory
schema_obj = schema.load_schema("/path/to/schema")

# Load from JSON file
schema_obj = schema.load_schema("/path/to/schema.json")
```

The function is **cached** (`@lru_cache`), so repeated calls with the same path return the cached result.

**Returns:** A `Namespace` object (dict-like structure) with three main sections.

---

### Schema Structure and Navigation

The loaded schema contains three top-level sections:

#### **1. `schema.objects`**
Contains 12 sub-namespaces defining semantic terms:
- **`entities`** - Name-value pairs in filenames (e.g., `sub`, `ses`, `task`)
- **`metadata`** - Name-value pairs in JSON sidecars
- **`suffixes`** - Filename suffixes describing file contents (e.g., `bold`, `anat`)
- **`datatypes`** - Subdirectories organizing files by type (e.g., `anat`, `func`, `meg`)
- **`extensions`** - File extensions (e.g., `.nii.gz`, `.json`)
- **`columns`** - TSV file column definitions
- **`enums`** - Enumerated value definitions
- **`formats`** - Value format constraints
- **`modalities`** - Data recording categories
- **`common_principles`** - Universal BIDS terms

#### **2. `schema.rules`**
Defines constraints and validation requirements:
- **`rules.files`** - Filename requirements by datatype (e.g., `rules.files.raw.anat`)
- **`rules.sidecars`** - JSON metadata field specifications
- **`rules.checks`** - Validation rules with error codes
- **`rules.tabular_data`** - TSV column requirements

#### **3. `schema.meta`**
Provides metadata for interpreting the schema, including version information.

#### **Access Patterns**

```python
# Dot notation
version = schema.bids_version
schema_ver = schema.schema_version

# Bracket notation
version = schema['bids_version']

# Nested access
entity = schema.objects.entities.subject
metadata_field = schema.objects.metadata.FlipAngle
suffix = schema.objects.suffixes.bold

# Complex paths
rules = schema.rules.files.raw.anat
check_rule = schema.rules.checks.mri.PhasePartUnits
```

---

### API Functions

#### **Schema Loading & Export**
- **`load_schema(schema_path=None)`** - Load schema (cached). Path can be YAML directory, JSON file, or None for bundled default.
- **`export_schema(schema)`** - Serialize schema to JSON string. Returns the schema as a JSON string via `.to_json()`.
- **`dereference(schema)`** - Replace schema references with actual content (automatically called by `load_schema` for YAML sources).
- **`flatten_enums(schema)`** - Simplify enum structures by merging conditions.

#### **Validation & Inspection**
- **`validate_schema(schema)`** - Validate schema against BIDS metaschema.
- **`get_schema_validator()`** - Retrieve the JSON schema validator.
- **`filter_schema(schema, keyword)`** - Filter schema by keyword criteria.

#### **Rule Processing** (in `rules.py`)
- **`regexify_filename_rules(rule_group, schema, depth)`** - Convert filename rules to regex patterns.
- **`regexify_all()`** - Create full path regexes for all BIDS specification files; returns (patterns, schema).

#### **Version Utilities**
- **`_get_schema_version()`** - Extract schema format version from `SCHEMA_VERSION` file.
- **`_get_bids_version()`** - Extract BIDS specification version from `BIDS_VERSION` file.

### Deprecations & Requirement Levels

Deprecated elements in the schema are marked with a `deprecated` level field. The BIDS schema recognizes four requirement levels:
- **REQUIRED** - Must be present
- **RECOMMENDED** - Should be present  
- **OPTIONAL** - May be present
- **DEPRECATED** - Previously required/recommended but no longer

Validators should treat deprecated elements as warnings rather than errors.

---

### Version Support

- **Single Bundled Version**: Each `bidsschematools` release includes one specific BIDS schema version bundled with it.
- **Version Selection**: To work with different BIDS versions (1.8, 1.9, 2.0, etc.), you can:
  1. Install different versions of `bidsschematools` (each bundles a corresponding BIDS schema)
  2. Load schema from external sources: `schema.load_schema("https://bids-specification.readthedocs.io/en/v1.8.0/schema.json")`
  3. Load from a custom schema directory with a specific BIDS version

The schema files are accessed through symlinks in `tools/schemacode/src/bidsschematools/data/`:
- `schema/` - Symlink to actual schema directory
- `metaschema.json` - Symlink to metaschema for validation

### Key Characteristics

**Lightweight:** Minimizes dependencies; serves as the reference parsing implementation  
**Dict-like:** Namespace objects support both dot notation and bracket notation for property access  
**Cached:** `load_schema()` caches results, so multiple calls with same path are efficient  
**CLI Available:** Command-line interface with `bst export` and other bindings

### How `bids-utils` Should Interact

1. **Schema Loading**: Use `bidsschematools.schema.load_schema()` to load the bundled or custom schema once at initialization
2. **Entity/Metadata Access**: Access definitions via `schema.objects.entities.<name>`, `schema.objects.metadata.<name>`, etc.
3. **Validation Rules**: Access validation constraints via `schema.rules.*` for filename and sidecar rules
4. **Rule Processing**: Use `bidsschematools.rules.regexify_*()` functions for filename validation regex generation
5. **Version Awareness**: Check `schema.bids_version` and `schema.schema_version` to understand which BIDS version is loaded
6. **Deprecation Handling**: Check for `deprecated` field in schema objects when accessing entities/metadata
7. **Custom Schema**: Support loading from custom schema paths for testing or custom BIDS configurations

### Important Notes for Integration

- The schema is a **read-only resource** - don't attempt to modify the loaded schema object
- Use **caching effectively** - load the schema once and pass it around rather than reloading
- **Version mismatch awareness** - document which version of bidsschematools (and thus BIDS schema) your tool expects
- **Dereferencing happens automatically** for YAML sources but not JSON sources
- The `Namespace` type is **dict-compatible** but also supports attribute access, making it flexible for navigation

### 5. WHAT CAN BE REUSED vs. NEEDS TO BE BUILT FRESH

#### **STRONGLY REUSABLE:**

1. **Migration Registry Framework** (from PR #2282)
   - Decorator-based registration is clean, extensible, and already tested
   - Can be directly imported/extended in bids-utils
   - Dry-run infrastructure already in place

2. **bidsschematools Schema Access APIs**
   - `load_schema()` for loading versioned schemas
   - `Namespace` for traversing schema definitions
   - Rule processing for regex validation
   - All deprecation metadata is machine-readable and queryable

3. **BIDS Schema Definition Files**
   - `deprecations.yml` is the authoritative source for all deprecations
   - `metadata.yaml` contains field definitions with replacement guidance
   - Both are parseable YAML that can drive migrations programmatically
   - **Design requirement FR-016/FR-017 explicitly demand** deriving all deprecation knowledge from these files, not hardcoding

4. **Dry-run and JSON Output Infrastructure**
   - PR #2282 already implements safe JSON operations
   - Dry-run mode with preview output is functional
   - Can be extended with `--json` formatted output

#### **MUST BE BUILT FRESH FOR BIDS-UTILS:**

1. **File and Directory Rename Operations**
   - bidsschematools focuses on validation, not file manipulation
   - bids-utils needs comprehensive file/directory rename with sidecar discovery
   - Must handle: `.json`, `.bvec`, `.bval`, `.tsv` (events), and schema-defined extensions
   - Must support VCS awareness (git, git-annex, DataLad)

2. **`_scans.tsv` Patching**
   - Not implemented in existing tools
   - Must update filenames when files are renamed/moved
   - Must handle multi-subject/session hierarchies correctly

3. **`participants.tsv` / `participants.json` Management**
   - Subject/session rename operations require coordinated updates
   - Must validate consistency between JSON and TSV formats

4. **Metadata Aggregation/Segregation**
   - IP-freely prototype exists but not integrated
   - Must respect BIDS inheritance principle
   - Must handle partial file absence (don't aggregate if metadata is missing from some subjects)

5. **File System Operations with VCS Awareness**
   - Must detect and use `git mv` instead of `os.rename`
   - Must support git-annex content management
   - Must support rollback/atomic operations

6. **Dataset Merging/Splitting Logic**
   - Beyond scope of bidsschematools
   - Requires conflict resolution strategies
   - Requires session/subject management

#### **PARTIALLY AVAILABLE:**

1. **Deprecation Application**
   - PR #2282 implements 3 migrations but only covers a subset of 1.x deprecations
   - Needs expansion to cover all deprecations mentioned in the design doc
   - Infrastructure is sound; implementations need to be added for each deprecation type

2. **Schema Version Targeting**
   - bidsschematools can load different schema versions
   - bids-utils design FR-010 requires explicit version selection and auto-detect from `BIDSVersion` field
   - This glue layer must be built
